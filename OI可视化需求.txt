本地 OI 可视化需求文档

一、目标

希望在本地查看港股期权某一条合约线的 Open Interest（OI）随时间变化，并辅助判断市场在某个行权价、某个到期日、某个方向（Call / Put）上的仓位集中情况。

该展示界面不是静态图，而是后续准备做成可交互的本地分析界面。


二、核心分析对象

分析对象不是“公司整体 OI”，而是“某一条具体期权合约线”。

一条合约线由以下字段唯一确定：

- 公司 company（界面展示与线名使用英文代码 / HKATS 代码，而不是数字股票代码）
- 行权价 strike
- 到期日 expiry
- 类型 type（Call 或 Put）

因此：

- 一条线 = 固定的 (company, strike, expiry, type)
- 线上的点 = 某个 date 对应的 oi 值
- 图中的时间序列 = oi(date)


三、图表定义（当前设想）

表 1：OI 图

- X 轴：时间 date
- Y 轴：OI 数量
- 一条线：某公司在某个 strike、某个 expiry、某个 Call / Put 下的 OI 随时间变化

表 2：Closing Price 图

- X 轴：时间 date
- Y 轴：Closing Price
- 一条线：某公司股票收盘价随时间变化

两个图的关系：

- Closing Price 与 OI 不放在同一个坐标系中
- 页面中使用上下两个图联动展示
- 两个图共享同一个时间轴
- 鼠标悬停时应吸附到最近日期
- 悬停时两个图都应同步定位到同一日期，便于观察价格变化与 OI 变化的对应关系


四、想要观察的市场信号

希望通过图来观察：

- 某条合约线的 OI 是否突然放大
- OI 放大发生在哪一天
- OI 放大之后是否持续累积
- 某个 Put / Call 的特定 strike 是否形成仓位集中
- 某个关键价位是否被市场当作重要支撑 / 压力参考

示例理解：

- 如果在周五看到“150、下个月到期、Put 的 OI 激增”，则说明市场在这条 Put 合约上集中建立仓位
- 进一步的主观解读可能是：市场认为到该到期日时，价格大幅跌破该线的概率或意愿相对有限

注意：上面属于分析解释，不等同于数据库中的直接字段，需要结合图表与主观判断完成。


五、数据来源与当前数据模型理解

当前数据表为 options_daily。

该表中与本需求直接相关的字段包括：

- date
- stock_code
- closing_price
- expiry
- strike
- type
- volume
- oi
- oi_change

表中一条记录表示：

- 某一天
- 某公司
- 某到期日
- 某行权价
- 某种类型（Call / Put）

对应的一条期权合约数据。

因此 options_daily 可以直接支持“单条合约线的 OI 时间序列”查询。


六、后续界面方向（尚未开始实现）

后续希望做一个本地交互界面，用于：

- 初始状态图表为空，不预置任何合约线
- 通过“添加一条线”的方式逐步加入要观察的合约线
- 选择公司
- 选择到期日
- 选择行权价
- 选择 Call / Put
- 展示对应合约线的 OI 时间序列
- 可能支持鼠标悬停、日期吸附、显示辅助信息

其中，“添加一条线”的交互规则当前明确为：

- 页面生成或初始化时，先从数据库一次性读取可选项所需的元数据，并缓存到页面侧
- “添加一条线”时，优先使用页面缓存中的可选项数据，不在每次加线时重新遍历数据库
- 先选择公司（界面中应展示英文代码 / HKATS 代码）
- 然后根据缓存中该公司实际存在的数据，动态生成可选的行权价 strike 列表
- 同时根据缓存中该公司实际存在的数据，动态生成可选的到期日 expiry 列表
- 可选项范围完全来自数据库现有记录，不预先写死，不事先限定固定范围
- 用户选择完成后，图中新增一条由 (company, strike, expiry, type) 唯一确定的合约线

此外，需要提供一个 sync 按钮：

- 用户点击 sync 时，再重新读取数据库中的元数据
- 重新生成页面中的可选项缓存
- 刷新页面内可供选择的公司、到期日、行权价等选项
- 对图中已经添加的线，也要重新读取其对应的数据
- 如果数据库中的 OI 或其他相关数值发生变化，图上的线位置也要同步更新
- sync 后图表需要基于最新数据重新渲染
- sync 是显式操作，不要求页面在每次交互时自动重新扫库

目前尚未进入实现阶段，本文件仅用于整理需求。


六点五、图上已添加线的控制能力

当前已确认：

- 已添加的线需要支持“隐藏但不删除”
- 已添加的线需要支持“删除”

当前暂不要求：

- 不要求用户自定义线颜色
- 不要求用户修改线名


六点六、Closing Price 与线名显示

当前已确认：

- 图中除了 OI 合约线外，还需要展示股票 Closing Price 随时间变化
- 引入 Closing Price 的目的，是辅助判断暴涨、暴跌等价格事件与某个价位 OI 变化之间的关系
- Closing Price 与 OI 不应混在同一张图里，而应拆成上下两个联动图
- 默认线名格式采用：英文代码-到期日-类型-行权价
- 例如：POP-29APR26-Call-150.5
- 这里的“代码”明确指英文代码 / HKATS 代码，不是数字股票代码


七、当前已明确与未明确事项

已明确：

- 核心对象是“单条合约线”的 OI，而不是公司整体 OI
- 一条线的唯一标识是 (company, strike, expiry, type)
- 图的核心是时间序列：date -> oi
- 页面中还需要一张独立的 Closing Price 时间序列图，用于辅助解释 OI 变化
- Closing Price 图与 OI 图需要时间轴联动
- 所有内容都希望在本地运行
- 图表初始状态为空，不默认展示任何线
- 线的可选参数需要根据数据库实际内容动态生成，而不是预设固定范围
- 页面初始化时应一次性读取并缓存可选项元数据，而不是每次加线都重新查数据库
- 需要提供手动 sync 机制，用于重新读取数据库并刷新页面缓存
- sync 不仅刷新可选项，还要刷新图上已存在线的数据，并使图形位置更新到最新数值
- 已添加的线需要支持隐藏与删除
- 第一版暂不支持自定义线颜色和自定义线名
- 界面展示与默认线名使用英文代码 / HKATS 代码，而不是数字股票代码

未明确：

- 第一版是否只看 OI，还是同时叠加 Closing Price
- 如果叠加 Closing Price，是放在同一张图还是作为辅助视图
- 是否允许同时勾选多条合约线进行对比
- “添加一条线”时，选择顺序是否固定为 公司 -> 到期日 -> 行权价 -> Call/Put，还是允许任意顺序联动筛选
- 时间范围选择是自由输入、预设区间，还是两者都要
- tooltip 中最终需要显示哪些字段
